# v0.4.0 — Limpar SPI antes que cole

Objetivo: corrigir anti-patterns no SPI enquanto ainda há poucos consumidores
externos. Tudo aqui é breaking; cada item entra como `feat!`, `refactor!` ou
`chore!`. Execute na ordem listada.

---

## Prompt 1 — Remover `OutboxDialectProvider#priority()`

**Branch:** `refactor/jdbc-drop-provider-priority`
**Commit prefix:** `refactor!(outbox-jdbc):`

**Contexto.** Em `outbox-jdbc/src/main/java/io/github/lobofoltran/outbox/jdbc/spi/OutboxDialectProvider.java`
existe um método `default int priority() { return 0; }` usado por
`DialectAutoDetector` para resolver empates quando múltiplos providers
reportam `supports(metaData) == true`. Esse padrão cria "leilões" silenciosos:
um provider terceiro com prioridade maior toma o lugar do builtin sem ninguém
perceber. É um anti-pattern conhecido em SPIs Java.

**Tarefa.**

1. Remover o método `priority()` de `OutboxDialectProvider`.
2. Em `DialectAutoDetector.detect(Connection)`, ao encontrar ≥2 providers cujo
   `supports(...)` retorne `true`, lançar
   `OutboxConfigurationException` com mensagem indicando os providers que
   bateram e instruindo a passar `JdbcOutbox.builder().dialect(...)`
   explicitamente.
3. Atualizar testes em `outbox-jdbc/src/test/java/.../internal/DialectAutoDetector*Test.java`
   para cobrir o novo comportamento: 0 matches → erro existente (não muda),
   1 match → ok, ≥2 matches → `OutboxConfigurationException`.
4. Remover qualquer override de `priority()` em testes ou no
   `PostgresDialectProvider`.
5. Atualizar Javadoc da interface explicando a nova regra "fail fast on
   ambiguous match".

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- TCK não regride.
- Mensagem do erro de ambiguidade cita ao menos os FQNs dos providers que
  conflitaram.

**Commit body deve conter `BREAKING CHANGE:`** explicando que dialetos
terceiros que dependiam de `priority()` devem ser desambiguados via
`builder.dialect(...)`.

## Prompt 2 — Remover `DialectCapability.BATCH_INSERT`

**Branch:** `refactor/jdbc-drop-batch-insert-capability`
**Commit prefix:** `refactor!(outbox-jdbc):`

**Contexto.** O enum `DialectCapability` (em `outbox-jdbc/src/main/java/.../spi/DialectCapability.java`)
lista `BATCH_INSERT` entre as capacidades opcionais. Toda JDBC 4+ suporta
`addBatch`/`executeBatch` — esse valor de enum é dead-weight e exporta uma
falsa decisão de design.

**Tarefa.**

1. Remover o valor `BATCH_INSERT` de `DialectCapability`.
2. Remover seu uso em `PostgresDialect.capabilities()` e em qualquer outro
   `OutboxDialect`.
3. Atualizar `OutboxDialectContractTest` (módulo `outbox-tck`) e qualquer
   teste que cite `BATCH_INSERT` — assertivas de "batch funciona" passam a
   ser incondicionais.
4. Atualizar Javadoc do enum.

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- Grep no repo por `BATCH_INSERT` retorna zero hits.

**`BREAKING CHANGE:`** dialetos terceiros que referenciavam
`DialectCapability.BATCH_INSERT` (improvável fora de testes) devem remover a
referência.

## Prompt 3 — Mover `IdGenerator` para o package `spi`

**Branch:** `refactor/jdbc-move-idgenerator-to-spi`
**Commit prefix:** `refactor!(outbox-jdbc):`

**Contexto.** `io.github.lobofoltran.outbox.jdbc.IdGenerator` é um ponto de
extensão (FunctionalInterface usado pelo `JdbcOutbox.Builder#idGenerator`).
Todo o resto do SPI vive em `io.github.lobofoltran.outbox.jdbc.spi`. A
inconsistência confunde consumidores e dificulta a regra "tudo o que é SPI
está em `.spi`".

**Tarefa.**

1. Mover `IdGenerator` para `io.github.lobofoltran.outbox.jdbc.spi`.
2. `UuidV7Generator` (em `internal`) continua onde está; só atualizar o
   import.
3. Atualizar `module-info.java` se necessário (o package `spi` já está
   exportado, então deve apenas continuar).
4. Atualizar todos os imports no repo (inclui `JdbcOutbox`, testes,
   `outbox-spring`).
5. Atualizar `JdbcOutboxBuilderCustomizer` Javadoc que cita o FQN antigo.

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- Nenhuma referência ao FQN antigo `...jdbc.IdGenerator` no código (somente
  no Javadoc do `@since` se for relevante manter histórico).

**`BREAKING CHANGE:`** consumidores que `import` o FQN antigo precisam
re-importar do novo package.

## Prompt 4 — UTF-8 length sem alocação em `PostgresDialect.validate`

**Branch:** `perf/jdbc-utf8-length-no-alloc`
**Commit prefix:** `perf(outbox-jdbc):`

**Contexto.** Em `PostgresDialect.requireMaxBytes(String, String, int)` o
código faz `value.getBytes(StandardCharsets.UTF_8).length` — aloca um `byte[]`
descartável só para medir o tamanho. Isso roda por evento, por campo
validado (5 campos), no caminho transacional crítico.

**Tarefa.**

1. Substituir a implementação por uma rotina que conta bytes UTF-8
   diretamente sobre `String.charAt(int)`, sem alocação:
   - char `< 0x80` → 1 byte
   - char `< 0x800` → 2 bytes
   - surrogate pair → 4 bytes (avança 2 chars)
   - resto → 3 bytes
2. Implementar como método estático `package-private` em
   `dialect.postgres` (não exportar, é detalhe de implementação). Nome
   sugerido: `Utf8Length.of(String)`.
3. Adicionar teste unitário cobrindo: ASCII puro, multibyte (1, 2, 3 bytes),
   surrogate pair (emoji), string vazia.
4. Garantir que o resultado bate exatamente com
   `s.getBytes(UTF_8).length` para uma amostra parametrizada.

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- JMH não é obrigatório, mas comentar no PR a economia esperada (5 alocs
  por evento × N eventos).
- Mutation score do PostgresDialect não regride.

Sem breaking change.

## Prompt 5 — `Outbox.publishAll`: contrato all-or-nothing + TCK

**Branch:** `feat/core-publishall-atomic-contract`
**Commit prefix:** `feat(outbox-core):` (sem `!` — só clarifica contrato)

**Contexto.** Hoje o Javadoc de `Outbox.publishAll` diz "se algum write
falhar, a TX não é commitada por esta lib". Isso é vago: callers assumem
all-or-nothing (que é o que PG entrega via `executeBatch`), mas o contrato
não promete. Precisamos formalizar e cobrir via TCK.

**Tarefa.**

1. Em `outbox-core/.../Outbox.java`, expandir o Javadoc de `publishAll`
   explicitando: "Implementações DEVEM garantir que, em caso de falha,
   nenhum dos eventos do lote tenha sido persistido na transação corrente
   do caller. Implementações que não consigam garantir essa propriedade
   DEVEM cair para publish individual via o método default."
2. Em `outbox-tck/.../OutboxDialectContractTest.java`, adicionar teste:
   - Publica lote de 3 eventos onde o 2º viola algum invariante (e.g.
     `aggregateType` excedendo width do dialeto via
     `OutboxDialect.validate`).
   - Espera `OutboxException` (qualquer subtipo apropriado).
   - Após `rollback()`, conta linhas na tabela: deve ser 0.
3. Adicionar IT correspondente em `outbox-jdbc` se ainda não houver
   (verificar `JdbcOutboxBatchIT`).
4. Adicionar nota no Javadoc de `JdbcOutbox.publishAll` referenciando o
   contrato.

**Aceitação.**

- Teste novo passa com `JdbcOutbox`/`PostgresDialect`.
- TCK roda contra PG em `PostgresDialectContractIT` sem regredir.

Sem breaking change — clarificação de contrato existente.

## Prompt 6 — `OutboxAutoConfiguration`: composição inline em vez de `BeanPostProcessor`

**Branch:** `refactor/spring-inline-decoration`
**Commit prefix:** `refactor(outbox-spring):`

**Contexto.** Em `OutboxAutoConfiguration` os decorators `MeteredOutbox` e
`TracedOutbox` são aplicados via dois `BeanPostProcessor` ordenados
(`Ordered.LOWEST_PRECEDENCE - 200` e `- 100`). Isso:

- Embrulha **qualquer** bean `Outbox` no contexto, inclusive os definidos
  pelo usuário (com guard `!(bean instanceof MeteredOutbox)` que não cobre
  todos os casos).
- Requer comentário extenso explicando precedence.
- Dispara em todo bean durante refresh.

Composição inline no `@Bean` factory resolve tudo isso e mantém a ordem
canônica `TracedOutbox(MeteredOutbox(JdbcOutbox))`.

**Tarefa.**

1. Remover as classes internas `MetricsBeanPostProcessor` e
   `TracingBeanPostProcessor` e suas `@Configuration` aninhadas
   (`MetricsConfiguration`, `TracingConfiguration`).
2. No método `@Bean public Outbox outbox(...)`, receber adicionalmente
   `ObjectProvider<MeterRegistry>` e `ObjectProvider<OpenTelemetry>`.
3. Após `builder.build()`, compor:
   - Se Micrometer disponível **e** `properties.metrics().enabled()` →
     envolver em `MeteredOutbox`.
   - Se OTel disponível **e** `properties.tracing().enabled()` → envolver
     o resultado em `TracedOutbox`.
4. Manter `@ConditionalOnClass({MeterRegistry.class, MeteredOutbox.class})`
   e `@ConditionalOnClass({Tracer.class, TracedOutbox.class})` como guards
   estáticos sobre a presença das classes — usar `ClassUtils.isPresent` se
   necessário, ou separar em métodos `@Bean` distintos retornando o
   delegate decorado (escolha o que ficar mais legível).
5. Atualizar `OutboxAutoConfigurationTest` para:
   - Validar que beans `Outbox` definidos pelo usuário **não** são
     decorados automaticamente (mudança semântica — documentar).
   - Validar ordem `TracedOutbox` envolvendo `MeteredOutbox`.
6. Atualizar o Javadoc da classe explicando a nova semântica e referenciar
   ADR-0017 (atualizar a ADR se necessário, em PR separado de docs).

**Aceitação.**

- Spring Boot IT (`OutboxSpringBootIT`, `OutboxOtelSpringBootIT`) passa.
- Métricas e spans continuam emitidos exatamente uma vez por publish.

**Atenção.** Esta mudança altera comportamento sutil para usuários que
definem `@Bean Outbox` próprio e contavam com auto-decoração. Documentar
no commit body sem `BREAKING CHANGE:` (é mudança de comportamento de
autoconfig, não de API).

## Prompt 7 — `schema_version` — escolher destino

**Branch:** `feat/schema-version-fix`
**Commit prefix:** `feat!(outbox-schema):` se mover; `feat(outbox-jdbc):` se popular

**Contexto.** A DDL `outbox-publisher.sql` inclui
`schema_version SMALLINT NOT NULL DEFAULT 1`, mas o INSERT do
`PostgresDialect` nunca menciona essa coluna. Toda row sai com `1` por
DEFAULT. Há duas saídas válidas; escolher **uma** e executar.

**Opção A — mover para relay extension.**

1. Remover `schema_version` de `outbox-publisher.sql`.
2. Adicionar `ADD COLUMN IF NOT EXISTS schema_version SMALLINT NOT NULL DEFAULT 1`
   em `outbox-relay-extension.sql`.
3. Justificar em comentário do SQL: "schema_version é concern do
   consumer/relay, não do publisher".
4. Atualizar AGENTS.md > Table contract.

**Opção B — popular explicitamente.**

1. Adicionar constante `SCHEMA_VERSION = 1` (ou similar) no
   `PostgresDialect`.
2. Incluir a coluna no `INSERT` e bindar no `PostgresOutboxInsert`.
3. Documentar no Javadoc do dialeto que valores >1 sinalizariam evolução
   de schema futura.

**Recomendação.** Opção A — mantém disciplina "publisher não conhece
relay". Mas a decisão é do reviewer humano; o agente deve abrir o PR com
**uma** das duas e justificar a escolha no body do PR.

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- Se opção A: `SchemaResourceTest` continua validando os dois SQLs.
- Se opção B: IT verifica `schema_version = 1` em row recém-inserida.

**Se opção A: `BREAKING CHANGE:`** adopters que aplicaram o publisher
DDL precisam aplicar a relay extension para manter a coluna (ou aceitar
o `DROP COLUMN`, dependendo da estratégia de retenção da migration).

## Prompt 9 — ADRs + NON-GOALS.md

**Branch:** `docs/v0.4.0-adrs-and-non-goals`
**Commit prefix:** `docs:`

**Tarefa.**

1. Criar `docs/adr/0018-destination-and-content-type-in-core.md` (template em
   `docs/adr/template.md`):
   - Contexto: `destination` e `contentType` em `OutboxEvent` são leituras
     de envelope de broker, não de domínio.
   - Decisão: aceitos em core como concessão consciente; documentado para
     que revisores futuros não os "limpem" achando que é vazamento
     acidental.
   - Consequências: callers sem broker (CDC-puro) tratam ambos como
     metadados livres.
2. Criar `docs/adr/0019-xa-and-distributed-tx-are-callers-concern.md`:
   - Decisão: a SDK não declara nem rejeita TX distribuída/XA.
     `ConnectionSupplier` é opaco; se o caller usar `XADataSource`,
     funciona; correção é responsabilidade do caller.
3. Criar `NON-GOALS.md` na raiz com a lista canônica:
   - sem `publishAsync` / reactive
   - sem polling / scheduler / relay
   - sem consumer API
   - sem broker connector (Kafka, SQS, …)
   - sem retry built-in
   - sem JPA bridge
   - sem Flyway/Liquibase bundled
   - sem serializer de payload
   - sem `OutboxTemplate`
   - sem auto-discovery do `Outbox` via ServiceLoader
4. Atualizar `CONTRIBUTING.md` referenciando `NON-GOALS.md`: PRs que
   pisarem nessa lista são rejeitados sem revisão técnica.

**Aceitação.**

- Os três arquivos existem e estão referenciados de `README.md` e
  `AGENTS.md`.
- ADRs seguem o template e estão linkadas no índice do `docs/adr/`.

Sem breaking change.
