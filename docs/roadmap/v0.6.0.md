# v0.6.0 — Validar SPI com segundo dialeto

**Pré-requisito.** Este release só faz sentido quando **um adopter real**
pede MySQL (ou outro DB). Não execute preventivamente. Manter `outbox-jdbc`
com PG-only enquanto não há demanda concreta é mais saudável do que ter
SPI desnecessariamente generalizado.

Quando a demanda chegar, execute os prompts abaixo em ordem.

---

## Prompt 1 — Spike: implementar MySQL 8 dialect num branch exploratório

**Branch:** `spike/mysql-dialect`
**Commit prefix:** N/A (spike — não merge direto)

**Objetivo.** Não é entregar; é descobrir onde o SPI quebra.

**Tarefa.**

1. Criar `outbox-jdbc/src/main/java/.../dialect/mysql/MySqlDialect.java`
   implementando `OutboxDialect` contra MySQL 8.
   - INSERT idempotente: `INSERT IGNORE INTO ...` (variante diferente
     do PG `ON CONFLICT`).
   - JSON column: MySQL tem `JSON` nativo; binding como `String` funciona.
   - UUID: MySQL não tem tipo nativo; armazenar como `BINARY(16)` ou
     `CHAR(36)`. Escolha precisa entrar em ADR.
   - Timestamp: `TIMESTAMP(6)` ou `DATETIME(6)`; tradeoff de timezone
     awareness.
2. Adicionar `MySqlDialectProvider`.
3. Rodar o TCK (`OutboxDialectContractTest`) contra MySQL via
   Testcontainers.
4. **Documentar no PR description** cada ponto onde o SPI atual:
   - foi suficiente,
   - precisou de capability nova,
   - precisou de método novo,
   - forçou contorção feia no dialeto.

**Aceitação do spike.** PR description responde: "este SPI cabe MySQL com
ou sem mudança? Que mudança?"

**Não fazer merge.** O spike informa o próximo prompt.

---

## Prompt 2 — ADR-0020: evolução do SPI baseado no spike

**Branch:** `docs/adr-0020-spi-evolution`
**Commit prefix:** `docs:`

**Tarefa.**

1. Criar `docs/adr/0020-spi-evolution-after-mysql-spike.md`.
2. Documentar o que foi aprendido no spike (Prompt 1 desta release).
3. Decidir e justificar:
   - Que métodos `default` adicionar ao `OutboxDialect`?
   - Que novos `DialectCapability` precisam existir?
   - Que mudanças no `OutboxInsert`?
   - Algum método existente precisa de signature aditivo?
4. Verificar se **alguma** mudança proposta é breaking. Se sim, este
   ainda é o release pra fazer (pré-1.0).

**Aceitação.** ADR aceita, linkada no índice.

## Prompt 3 — Aditivos ao SPI (não-breaking quando possível)

**Branch:** `feat/jdbc-spi-additions-for-mysql`
**Commit prefix:** `feat(outbox-jdbc):` ou `feat!(outbox-jdbc):` se a ADR-0020 decidiu por breaking

**Tarefa.**

1. Aplicar exatamente o que a ADR-0020 prescreveu.
2. Manter `PostgresDialect` 100% funcional sem mudança de comportamento.
3. Cobertura: cada novo método `default` tem unit test
   verificando default vs override.

**Aceitação.**

- TCK passa contra PG.
- Coverage estável.

---

## Prompt 4 — Promover MySQL dialect a código de produção

**Branch:** `feat/jdbc-mysql-dialect`
**Commit prefix:** `feat(outbox-jdbc):`

**Tarefa.**

1. Pegar o spike (Prompt 1), reescrever conforme o SPI atualizado pela
   ADR-0020 + Prompt 3.
2. `MySqlDialect` + `MySqlDialectProvider` em
   `io.github.lobofoltran.outbox.jdbc.dialect.mysql` (não exportado).
3. Registrar no `module-info.java` via `provides ... with ...`.
4. Atualizar `outbox-schema` com `outbox-publisher-mysql.sql` na pasta
   `sql/mysql/`.
5. Criar `MySqlDialectContractIT` em `outbox-tck/src/test/java/...`
   estendendo `OutboxDialectContractTest`.
6. Atualizar `JdbcOutboxDialectAutoDetectionIT` cobrindo seleção entre
   PG e MySQL.
7. Documentar no README: "Officially supported: PostgreSQL 16, MySQL 8".

**Aceitação.**

- TCK passa contra PG e MySQL.
- Coverage nos targets configurados.

Sem breaking change (novo dialeto é puramente aditivo).

## Prompt 5 — Validar que o TCK serve a alguém de fora

**Branch:** `chore/tck-evaluation`
**Commit prefix:** `chore:`

**Tarefa.**

1. Confirmar com pelo menos um stakeholder externo (issue/comentário no
   GitHub) que pretende usar o TCK para um dialeto terceiro
   (e.g. Oracle, MSSQL).
2. Se sim: TCK fica como módulo público, prepare para publicar em 1.0.
3. Se não: criar issue de tracking para reavaliar em 6 meses, e marcar
   o `outbox-tck` como "internal until 1.0" em README.
