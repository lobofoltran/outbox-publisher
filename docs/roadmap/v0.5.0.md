## Prompt 1 — Extrair `OutboxHealthIndicator` para `outbox-spring-actuator`

**Branch:** `feat/spring-actuator-module`
**Commit prefix:** `feat!(outbox-spring):` (remover bean) + `feat(outbox-spring-actuator):` (novo módulo)

**Contexto.** `OutboxHealthIndicator` hoje vive em `outbox-spring` e é
default-on quando Actuator está no classpath. O probe abre conexão e roda
SQL contra a tabela alvo a cada poll — sob carga, contribui para starvation
de pool, e o valor agregado é marginal (checa schema/permissão, não saúde
do publisher).

**Tarefa.**

1. Criar módulo Maven `outbox-spring-actuator`:
   - `groupId`, `parent`, `version` iguais aos demais módulos.
   - Depende de `outbox-spring` e
     `spring-boot-starter-actuator` (scope `provided`).
   - JPMS: automatic module (consistente com `outbox-spring`).
2. Mover `OutboxHealthIndicator` e `OutboxHealthIndicatorTest`/IT para o
   novo módulo, package
   `io.github.lobofoltran.outbox.spring.actuator`.
3. Mover a `HealthConfiguration` aninhada de `OutboxAutoConfiguration` para
   uma nova `@AutoConfiguration` no novo módulo. Atualizar
   `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
   nos dois módulos.
4. Atualizar `outbox-bom` adicionando o novo artefato.
5. Remover `OutboxProperties.Health` de `OutboxProperties` — mover para um
   novo record `OutboxActuatorProperties` no novo módulo (ou manter prefixo
   `io.github.lobofoltran.outbox.health` por compatibilidade).
6. Atualizar `pom.xml` raiz, `release-please-config.json`, e a documentação
   (README, examples).

**Aceitação.**

- `./mvnw -B -ntp -Pquality verify` passa.
- Apps que querem o health indicator agora declaram
  `outbox-spring-actuator` explicitamente.
- Apps que **não** declaram o novo módulo: sem health bean, sem mudança em
  outros comportamentos.

**`BREAKING CHANGE:`** o health indicator deixa de ser ativado por
`outbox-spring` + Actuator no classpath; adicionar `outbox-spring-actuator`
para mantê-lo.

## Prompt 2 — `JdbcOutbox.Builder#queryTimeout(Duration)`

**Branch:** `feat/jdbc-query-timeout`
**Commit prefix:** `feat(outbox-jdbc):`

**Contexto.** Hoje nada limita o tempo de um INSERT pendurado em lock no
índice. Em produção, isso pode segurar conexões indefinidamente.

**Tarefa.**

1. Adicionar `queryTimeout(Duration)` em `JdbcOutbox.Builder`, default
   `null` (sem timeout — comportamento atual).
2. Propagar para o `PreparedStatement.setQueryTimeout(int seconds)` no
   `PostgresOutboxInsert.bind` ou no `PostgresDialect.prepareInsert`
   (depende de onde fica mais limpo — provavelmente passar o timeout via
   `TableRef` é errado; melhor passar para `OutboxDialect.prepareInsert`
   como argumento adicional ou via campo no handle).
3. Aditar default no SPI: `OutboxDialect.prepareInsert(Connection, TableRef, Duration timeout)`
   com `default` redirecionando para a assinatura antiga, marcada como
   deprecated. Em 1.0 a antiga será removida.
4. Adicionar IT cobrindo: tabela com lock segurado por outra TX,
   timeout=1s, publish deve falhar com `OutboxTransientException`
   (verifica SQLState `57014` que é cancelamento por timeout no PG).
5. Documentar no Javadoc do builder e do `OutboxDialect`.
6. Configuração no Spring: `OutboxProperties#queryTimeout` (Duration),
   bound via `JdbcOutboxBuilderCustomizer` interno.

**Aceitação.**

- IT de timeout passa contra PG.
- Apps que não setam timeout: nada muda.

Sem breaking change (default ao novo método mantém compat de SPI).

## Prompt 3 — `OutboxEvent.payloadView()` zero-copy

**Branch:** `feat/core-payload-view`
**Commit prefix:** `feat(outbox-core):`

**Contexto.** `OutboxEvent.payload()` clona em todo acesso por imutabilidade.
Decorators que precisam ler o payload (assinatura, criptografia,
validação externa) pagam clone. `payloadSize()` cobre o caso comum de
métrica; falta cobrir o caso "ler bytes sem clonar".

**Tarefa.**

1. Adicionar `ByteBuffer payloadView()` em `OutboxEvent` retornando
   `ByteBuffer.wrap(payload).asReadOnlyBuffer()`. Não cria cópia do
   array.
2. Javadoc deve deixar claro:
   - View é read-only.
   - Posição/limit/mark podem ser manipulados sem afetar o record.
   - Não compartilhar a view entre threads.
3. Atualizar Javadoc de `payload()` apontando para `payloadView()` como
   alternativa zero-copy quando o caller só precisa ler.
4. Atualizar `OutboxEventTest`: novo teste cobrindo `payloadView()` é
   read-only e reflete os bytes corretos.

**Aceitação.**

- Coverage não regride.
- Nenhuma mudança em comportamento existente.

Sem breaking change.

## Prompt 4 — Documentar capacity planning no README

**Branch:** `docs/readme-capacity-planning`
**Commit prefix:** `docs:`

**Tarefa.** Adicionar seção no README cobrindo:

- **WAL bloat:** outbox writes inflam WAL; pareie com retenção agressiva
  no relay (DELETE de rows SENT).
- **Índice parcial `idx_outbox_pending`:** é minúsculo sob operação
  saudável (poucos rows PENDING). Sem ele, polling vira full table scan.
- **Sizing inicial sugerido:** baseado em throughput
  (e.g. "10k events/s → ~1 GB WAL/h adicional, planeje checkpoint
  pressure").
- **Retenção:** o publisher não apaga; o relay/CDC apaga. Sem retenção,
  a tabela cresce indefinidamente.
- **UUIDv7 evita hot spot:** notar que o default já mitiga problema
  clássico de sequência.

Sem código mudando.

**Aceitação.** Seção `## Operating the outbox at scale` (ou similar)
existe no README com os pontos acima, com SQL exemplo para retenção.

## Prompt 6 — Mensagens de erro de `requireManualTransaction` mais ricas

**Branch:** `feat/jdbc-richer-autocommit-error`
**Commit prefix:** `feat(outbox-jdbc):`

**Contexto.** A mensagem hoje é genérica. Inclui guidance para Spring e
plain JDBC, mas não diz qual `DataSource`/pool está fornecendo a conexão
com autocommit ligado.

**Tarefa.**

1. Em `JdbcOutbox.requireManualTransaction`, ao detectar autocommit,
   incluir na mensagem:
   - `connection.getClass().getName()` (identifica o pool: Hikari,
     C3P0, raw driver, etc).
   - `connection.getMetaData().getDatabaseProductName()` + versão.
2. Capturar `SQLException` ao extrair metadata — não deixar mascarar o
   problema original.
3. Atualizar teste de unit cobrindo a nova mensagem.

**Aceitação.**

- Mensagem cita classe da conexão.
- Teste verifica formato.

Sem breaking change.
